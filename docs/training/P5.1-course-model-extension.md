# Phase 5.1 — Course Model Extension Decision Note

> **Status**: FROZEN | **Date**: 2026-03-13 | **Sprint**: P5.1

---

## Purpose

Document how Course-specific fields map to the existing `training_items` schema, resolve semantic overlaps, and confirm what was intentionally not implemented.

---

## A. Schema Changes Made

**No new columns were added.** All required Course fields already exist in `training_items`:

| Requested Field | Existing Column | Type | Notes |
|---|---|---|---|
| `duration_hours` | `duration_hours` | integer, nullable | Already existed |
| `competency_tag_ids` | `skill_term_ids` + `competency_domain_term_ids` | uuid[], nullable | Split across two columns by design |
| `certification_awarded` | `credential_eligible` | boolean, NOT NULL, default false | Semantic equivalent — see §D |
| `micro_assessment_enabled` | `micro_assessment` | boolean, NOT NULL, default false | Added in P5 migration |
| `cri_boost_value` | `cri_boost_value` | smallint, nullable | Added in P5 migration |

---

## B. Validation Rules Added

Course-specific validation in `src/lib/training/course-form-schema.ts`:

- `duration_hours`: integer, >= 1, nullable
- `skill_term_ids`: array of valid UUIDs (skills domain)
- `competency_domain_term_ids`: array of valid UUIDs (competency_domains domain)
- `credential_eligible`: boolean, required
- `micro_assessment`: boolean, required
- `cri_boost_value`: integer, 0–100, nullable
- No free-text competency tags permitted

- **Admin editor**: `/admin/training/new` and `/admin/training/:id/edit`

## D. Semantic Conflicts Resolved

### `certification_awarded` vs `credential_eligible`

**Decision**: `credential_eligible` is the canonical field.

The request's `certification_awarded` and the existing `credential_eligible` have identical semantics: "completing this item may result in a credential artifact." Rather than creating a duplicate boolean, the admin form labels this as "Certification Awarded" while writing to `credential_eligible`.

### `competency_tag_ids` (combined) vs `skill_term_ids` + `competency_domain_term_ids` (split)

**Decision**: Keep split columns as canonical.

The existing architecture separates skills (granular abilities) from competency domains (7 umbrella categories). Merging them into a single array would lose this semantic distinction. The admin form provides two separate multi-selects.

### `cri_boost_value` type: `numeric(5,2)` vs `smallint`

**Decision**: Keep `smallint` per the P5 frozen decision note.

The frozen architecture specifies `smallint`. Integer precision is sufficient for a 0–100 weight scale.
