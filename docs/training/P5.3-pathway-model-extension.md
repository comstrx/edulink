# Phase 5.3 — Pathway Model Extension Decision Note

> **Status**: FROZEN | **Date**: 2026-03-13 | **Sprint**: P5.3

---

## Purpose

Document how Pathway fields map to the existing `training_items` schema, resolve modeling gaps, and confirm what was intentionally not implemented.

---

## A. Schema Changes Made

Three new columns added to `training_items`:

| Field | Type | Nullable | Default | Purpose |
|---|---|---|---|---|
| `required_course_ids` | uuid[] | yes | '{}' | Ordered list of required course item IDs for the pathway journey |
| `milestones_json` | jsonb | yes | NULL | Structured milestone definitions (design metadata only) |
| `reflection_prompts_json` | jsonb | yes | NULL | Structured reflection prompt definitions (design metadata only) |

**Reused existing columns** (no new columns needed):

| Concept | Maps To | Notes |
|---|---|---|
| `target_competency_ids` | `skill_term_ids` + `competency_domain_term_ids` | Same columns used by courses; pathway uses them for target competencies |
| `cri_target_value` | `cri_target` | Already exists on training_items from Phase 5 base schema |

**GIN index** added on `required_course_ids` for efficient containment queries.

---

## B. Validation Rules Added

Pathway-specific validation in `src/lib/training/pathway-form-schema.ts`:

- `required_course_ids`: array of valid UUIDs; must reference type='course' items
- `cri_target`: numeric >= 0, max 999.99, nullable
- `milestones`: structured JSON array with validated schema:
  - `id`: required, unique within pathway
  - `title`: required
  - `order`: required, positive integer
  - `linked_course_ids`: must be subset of `required_course_ids`
- `reflection_prompts`: structured JSON array with validated schema:
  - `id`: required, unique within pathway
  - `prompt`: required
  - `order`: required, positive integer
  - `stage_id`: if provided, must reference existing milestone ID
- No free-text competency tags
- No user state stored in pathway metadata
- No self-referencing pathways

Application-layer enforcement:
- Course selector only shows `type = 'course'` items (via `useAvailableCourses()`)
- Milestone linked courses validated as subset of required courses
- Reflection prompt stage_id validated against milestone IDs

## D. Semantic Conflicts & Modeling Gaps

### 1. `target_competency_ids` reuses `skill_term_ids` + `competency_domain_term_ids`

**Decision**: Reuse existing columns rather than creating new pathway-specific columns.

Both courses and pathways use the same taxonomy domains (skills, competency_domains) for capability targeting. The semantic meaning differs slightly:
- Course: "what skills this course teaches"
- Pathway: "what competencies this pathway is designed to build"

The distinction is contextual (determined by `type`), not structural. Creating duplicate columns would be over-engineering.

### 2. `cri_target` already exists

**Decision**: Reuse the existing `cri_target` column on `training_items`.

This column was added in the base Phase 5 schema. It maps directly to the pathway's CRI readiness target. No new column needed.

### 3. JSONB vs relational for milestones/reflections

**Decision**: Use JSONB columns for both milestones and reflection prompts.

Milestones and reflection prompts are pathway-internal design metadata, not independently queryable entities. They don't need their own tables because:
- They are never queried independently of their parent pathway
- They don't have foreign key relationships to other tables
- Their cardinality is small (typically 3-7 milestones per pathway)
- Structured validation is enforced at the application layer via Zod

If milestones later need to become first-class entities (e.g., for completion tracking), they can be extracted to a `training_pathway_milestones` table.

### 4. Missing taxonomy domain for target segments

**Gap documented**: Same as Phase 5.2 — no canonical target-segment taxonomy exists yet.

## F. Architectural Decisions

### Pathway composition rules
- A pathway may only contain items where `training_items.type = 'course'`
- No package-inside-pathway
- No pathway-inside-pathway
- No arbitrary catalog nesting
- Enforced at application layer via `useAvailableCourses()` filter

### Pathway is design metadata, not execution
- `milestones_json` and `reflection_prompts_json` define the pathway structure
- They do NOT store user progress, submissions, or completion state
- Execution layers (progress, reflection, completion) will be built later

### CRI target is metadata, not scoring
- `cri_target` expresses intended readiness impact
- It is NOT a live score, not dynamically calculated, not user-specific
- CRI runtime belongs to the Intelligence layer

### Canonical data flow
```
Pathway Metadata (training_items where type = 'pathway')
        ↓
Required Courses (required_course_ids array)
        ↓
Milestones (milestones_json, design metadata)
        ↓
Reflection Prompts (reflection_prompts_json, design metadata)
        ↓
[Future] Pathway Enrollment
        ↓
[Future] Milestone Progress
        ↓
[Future] Reflection Submissions
        ↓
[Future] Pathway Completion
        ↓
[Future] CRI Score Update
```
